1 # moodycamel::ConcurrentQueue<T>
2 
3 An industrial-strength lock-free queue for C++.
4 
5 Note: If all you need is a single-producer, single-consumer queue, I have [one of those too][spsc].
6 
7 ## Features
8 
9 - Knock-your-socks-off [blazing fast performance][benchmarks].
10 - Single-header implementation. Just drop it in your project.
11 - Fully thread-safe lock-free queue. Use concurrently from any number of threads.
12 - C++11 implementation -- elements are moved (instead of copied) where possible.
13 - Templated, obviating the need to deal exclusively with pointers -- memory is managed for you.
14 - No artificial limitations on element types or maximum count.
15 - Memory can be allocated once up-front, or dynamically as needed.
16 - Fully portable (no assembly; all is done through standard C++11 primitives).
17 - Supports super-fast bulk operations.
18 - Includes a low-overhead blocking version (BlockingConcurrentQueue).
19 - Exception safe.
20 
21 ## Reasons to use
22 
23 There are not that many full-fledged lock-free queues for C++. Boost has one, but it's limited to objects with trivial
24 assignment operators and trivial destructors, for example. Intel's TBB queue isn't lock-free, and requires trivial constructors too.
25 There's many academic papers that implement lock-free queues in C++, but usable source code is
26 hard to find, and tests even more so.
27 
28 This queue not only has less limitations than others (for the most part), but [it's also faster][benchmarks].
29 It's been fairly well-tested, and offers advanced features like **bulk enqueueing/dequeueing**
30 (which, with my new design, is much faster than one element at a time, approaching and even surpassing
31 the speed of a non-concurrent queue even under heavy contention).
32 
33 In short, there was a lock-free queue shaped hole in the C++ open-source universe, and I set out
34 to fill it with the fastest, most complete, and well-tested design and implementation I could.
35 The result is `moodycamel::ConcurrentQueue` :-)
36 
37 ## Reasons *not* to use
38 
39 The fastest synchronization of all is the kind that never takes place. Fundamentally,
40 concurrent data structures require some synchronization, and that takes time. Every effort
41 was made, of course, to minimize the overhead, but if you can avoid sharing data between
42 threads, do so!
43 
44 Why use concurrent data structures at all, then? Because they're gosh darn convenient! (And, indeed,
45 sometimes sharing data concurrently is unavoidable.)
46 
47 My queue is **not linearizable** (see the next section on high-level design). The foundations of
48 its design assume that producers are independent; if this is not the case, and your producers
49 co-ordinate amongst themselves in some fashion, be aware that the elements won't necessarily
50 come out of the queue in the same order they were put in *relative to the ordering formed by that co-ordination*
51 (but they will still come out in the order they were put in by any *individual* producer). If this affects
52 your use case, you may be better off with another implementation; either way, it's an important limitation
53 to be aware of.
54 
55 My queue is also **not NUMA aware**, and does a lot of memory re-use internally, meaning it probably doesn't
56 scale particularly well on NUMA architectures; however, I don't know of any other lock-free queue that *is*
57 NUMA aware (except for [SALSA][salsa], which is very cool, but has no publicly available implementation that I know of).
58 
59 Finally, the queue is **not sequentially consistent**; there *is* a happens-before relationship between when an element is put
60 in the queue and when it comes out, but other things (such as pumping the queue until it's empty) require more thought
61 to get right in all eventualities, because explicit memory ordering may have to be done to get the desired effect. In other words,
62 it can sometimes be difficult to use the queue correctly. This is why it's a good idea to follow the [samples][samples.md] where possible.
63 On the other hand, the upside of this lack of sequential consistency is better performance.
64 
65 ## High-level design
66 
67 Elements are stored internally using contiguous blocks instead of linked lists for better performance.
68 The queue is made up of a collection of sub-queues, one for each producer. When a consumer
69 wants to dequeue an element, it checks all the sub-queues until it finds one that's not empty.
70 All of this is largely transparent to the user of the queue, however -- it mostly just works<sup>TM</sup>.
71 
72 One particular consequence of this design, however, (which seems to be non-intuitive) is that if two producers
73 enqueue at the same time, there is no defined ordering between the elements when they're later dequeued.
74 Normally this is fine, because even with a fully linearizable queue there'd be a race between the producer
75 threads and so you couldn't rely on the ordering anyway. However, if for some reason you do extra explicit synchronization
76 between the two producer threads yourself, thus defining a total order between enqueue operations, you might expect
77 that the elements would come out in the same total order, which is a guarantee my queue does not offer. At that
78 point, though, there semantically aren't really two separate producers, but rather one that happens to be spread
79 across multiple threads. In this case, you can still establish a total ordering with my queue by creating
80 a single producer token, and using that from both threads to enqueue (taking care to synchronize access to the token,
81 of course, but there was already extra synchronization involved anyway).
82 
83 I've written a more detailed [overview of the internal design][blog], as well as [the full
84 nitty-gritty details of the design][design], on my blog. Finally, the
85 [source][source] itself is available for perusal for those interested in its implementation.

87 ## Basic use
88 
89 The entire queue's implementation is contained in **one header**, [`concurrentqueue.h`][concurrentqueue.h].
90 Simply download and include that to use the queue. The blocking version is in a separate header,
91 [`blockingconcurrentqueue.h`][blockingconcurrentqueue.h], that depends on the first.
92 The implementation makes use of certain key C++11 features, so it requires a fairly recent compiler
93 (e.g. VS2012+ or g++ 4.8; note that g++ 4.6 has a known bug with `std::atomic` and is thus not supported).
94 The algorithm implementations themselves are platform independent.
95 
96 Use it like you would any other templated queue, with the exception that you can use
97 it from many threads at once :-)
98 
99 Simple example:
100 
101  #include "concurrentqueue.h"
102 
103  moodycamel::ConcurrentQueue<int> q;
104  q.enqueue(25);
105 
106  int item;
107  bool found = q.try_dequeue(item);
108  assert(found && item == 25);
109 
110 Description of basic methods:
111 - `ConcurrentQueue(size_t initialSizeEstimate)`
112  Constructor which optionally accepts an estimate of the number of elements the queue will hold
113 - `enqueue(T&& item)`
114  Enqueues one item, allocating extra space if necessary
115 - `try_enqueue(T&& item)`
116  Enqueues one item, but only if enough memory is already allocated
117 - `try_dequeue(T& item)`
118  Dequeues one item, returning true if an item was found or false if the queue appeared empty
119 
120 Note that it is up to the user to ensure that the queue object is completely constructed before
121 being used by any other threads (this includes making the memory effects of construction
122 visible, possibly via a memory barrier). Similarly, it's important that all threads have
123 finished using the queue (and the memory effects have fully propagated) before it is
124 destructed.
125 
126 There's usually two versions of each method, one "explicit" version that takes a user-allocated per-producer or
127 per-consumer token, and one "implicit" version that works without tokens. Using the explicit methods is almost
128 always faster (though not necessarily by a huge factor). Apart from performance, the primary distinction between them
129 is their sub-queue allocation behaviour for enqueue operations: Using the implicit enqueue methods causes an
130 automatically-allocated thread-local producer sub-queue to be allocated (it is marked for reuse once the thread exits).
131 Explicit producers, on the other hand, are tied directly to their tokens' lifetimes (and are also recycled as needed).
132 
133 Full API (pseudocode):
134 
135  # Allocates more memory if necessary
136  enqueue(item) : bool
137  enqueue(prod_token, item) : bool
138  enqueue_bulk(item_first, count) : bool
139  enqueue_bulk(prod_token, item_first, count) : bool
140 
141  # Fails if not enough memory to enqueue
142  try_enqueue(item) : bool
143  try_enqueue(prod_token, item) : bool
144  try_enqueue_bulk(item_first, count) : bool
145  try_enqueue_bulk(prod_token, item_first, count) : bool

160 ## Blocking version
161 
162 As mentioned above, a full blocking wrapper of the queue is provided that adds
163 `wait_dequeue` and `wait_dequeue_bulk` methods in addition to the regular interface.
164 This wrapper is extremely low-overhead, but slightly less fast than the non-blocking
165 queue (due to the necessary bookkeeping involving a lightweight semaphore).
166 
167 The only major caveat with the blocking version is that you must be careful not to
168 destroy the queue while somebody is waiting on it. This generally means you need to
169 know for certain that another element is going to come along before you call one of
170 the blocking methods.
171 
172 Blocking example:
173 
174  #include "blockingconcurrentqueue.h"
175 
176  moodycamel::BlockingConcurrentQueue<int> q;
177  std::thread producer([&]() {
178  for (int i = 0; i != 100; ++i) {
179  q.enqueue(i);
180  }
181  });
182  std::thread consumer([&]() {
183  for (int i = 0; i != 100; ++i) {
184  int item;
185  q.wait_dequeue(item);
186  assert(item == i);
187  }
188  });
189  producer.join();
190  consumer.join();
191 
192  assert(q.size_approx() == 0);

194 ## Advanced features
195 
196 #### Tokens
197 
198 The queue can take advantage of extra per-producer and per-consumer storage if
199 it's available to speed up its operations. This takes the form of "tokens":
200 You can create a consumer token and/or a producer token for each thread or task
201 (tokens themselves are not thread-safe), and use the methods that accept a token
202 as their first parameter:
203 
204  moodycamel::ConcurrentQueue<int> q;
205 
206  moodycamel::ProducerToken ptok(q);
207  q.enqueue(ptok, 17);
208 
209  moodycamel::ConsumerToken ctok(q);
210  int item;
211  q.try_dequeue(ctok, item);
212  assert(item == 17);
213 
214 If you happen to know which producer you want to consume from (e.g. in
215 a single-producer, multi-consumer scenario), you can use the `try_dequeue_from_producer`
216 methods, which accept a producer token instead of a consumer token, and cut some overhead.
217 
218 Note that tokens work with the blocking version of the queue too.
219 
220 When producing or consuming many elements, the most efficient way is to:
221 
222 1. Use the bulk methods of the queue with tokens
223 2. Failing that, use the bulk methods without tokens
224 3. Failing that, use the single-item methods with tokens
225 4. Failing that, use the single-item methods without tokens
226 
227 Having said that, don't create tokens willy-nilly -- ideally there would be
228 one token (of each kind) per thread. The queue will work with what it is
229 given, but it performs best when used with tokens.
230 
231 Note that tokens aren't actually tied to any given thread; it's not technically
232 required that they be local to the thread, only that they be used by a single
233 producer/consumer at a time.
234 
235 #### Bulk operations
236 
237 Thanks to the [novel design][blog] of the queue, it's just as easy to enqueue/dequeue multiple
238 items as it is to do one at a time. This means that overhead can be cut drastically for
239 bulk operations. Example syntax:
240 
241  moodycamel::ConcurrentQueue<int> q;
242 
243  int items[] = { 1, 2, 3, 4, 5 };
244  q.enqueue_bulk(items, 5);
245 
246  int results[5]; // Could also be any iterator
247  size_t count = q.try_dequeue_bulk(results, 5);
248  for (size_t i = 0; i != count; ++i) {
249  assert(results[i] == items[i]);
250  }
251 
252 #### Preallocation (correctly using `try_enqueue`)
253 
254 `try_enqueue`, unlike just plain `enqueue`, will never allocate memory. If there's not enough room in the
255 queue, it simply returns false. The key to using this method properly, then, is to ensure enough space is
256 pre-allocated for your desired maximum element count.
257 
258 The constructor accepts a count of the number of elements that it should reserve space for. Because the
259 queue works with blocks of elements, however, and not individual elements themselves, the value to pass
260 in order to obtain an effective number of pre-allocated element slots is non-obvious.
261 
262 First, be aware that the count passed is rounded up to the next multiple of the block size. Note that the
263 default block size is 32 (this can be changed via the traits). Second, once a slot in a block has been
264 enqueued to, that slot cannot be re-used until the rest of the block has completely been completely filled
265 up and then completely emptied. This affects the number of blocks you need in order to account for the
266 overhead of partially-filled blocks. Third, each producer (whether implicit or explicit) claims and recycles
267 blocks in a different manner, which again affects the number of blocks you need to account for a desired number of
268 usable slots.
269 
270 Suppose you want the queue to be able to hold at least `N` elements at any given time. Without delving too
271 deep into the rather arcane implementation details, here are some simple formulas for the number of elements
272 to request for pre-allocation in such a case. Note the division is intended to be arithmetic division and not
273 integer division (in order for `ceil()` to work).
274 
275 For explicit producers (using tokens to enqueue):
276 
277  (ceil(N / BLOCK_SIZE) + 1) * MAX_NUM_PRODUCERS * BLOCK_SIZ
278 
279 For implicit producers (no tokens):
280 
281  (ceil(N / BLOCK_SIZE) - 1 + 2 * MAX_NUM_PRODUCERS) * BLOCK_SIZE
282 
283 When using mixed producer types:
284 
285  ((ceil(N / BLOCK_SIZE) - 1) * (MAX_EXPLICIT_PRODUCERS + 1) + 2 * (MAX_IMPLICIT_PRODUCERS + MAX_EXPLICIT_PRODUCERS)) * BLOCK_SIZE
286 
287 If these formulas seem rather inconvenient, you can use the constructor overload that accepts the minimum
288 number of elements (`N`) and the maximum number of producers and consumers directly, and let it do the
289 computation for you.
290 
291 Finally, it's important to note that because the queue is only eventually consistent and takes advantage of
292 weak memory ordering for speed, there's always a possibility that under contention `try_enqueue` will fail
293 even if the queue is correctly pre-sized for the desired number of elements. (e.g. A given thread may think that
294 the queue's full even when that's no longer the case.) So no matter what, you still need to handle the failure
295 case (perhaps looping until it succeeds), unless you don't mind dropping elements.
296 
297 #### Exception safety
298 
299 The queue is exception safe, and will never become corrupted if used with a type that may throw exceptions.
300 The queue itself never throws any exceptions (operations fail gracefully (return false) if memory allocation
301 fails instead of throwing `std::bad_alloc`).
302 
303 It is important to note that the guarantees of exception safety only hold if the element type never throws
304 from its destructor, and that any iterators passed into the queue (for bulk operations) never throw either.
305 Note that in particular this means `std::back_inserter` iterators must be used with care, since the vector
306 being inserted into may need to allocate and throw a `std::bad_alloc` exception from inside the iterator;
307 so be sure to reserve enough capacity in the target container first if you do this.
308 
309 The guarantees are presently as follows:
310 - Enqueue operations are rolled back completely if an exception is thrown from an element's constructor.
311  For bulk enqueue operations, this means that elements are copied instead of moved (in order to avoid
312  having only some of the objects be moved in the event of an exception). Non-bulk enqueues always use
313  the move constructor if one is available.
314 - If the assignment operator throws during a dequeue operation (both single and bulk), the element(s) are
315  considered dequeued regardless. In such a case, the dequeued elements are all properly destructed before
316  the exception is propagated, but there's no way to get the elements themselves back.
317 - Any exception that is thrown is propagated up the call stack, at which point the queue is in a consistent
318  state.
319 
320 Note: If any of your type's copy constructors/move constructors/assignment operators don't throw, be sure
321 to annotate them with `noexcept`; this will avoid the exception-checking overhead in the queue where possible
322 (even with zero-cost exceptions, there's still a code size impact that has to be taken into account).
323 
324 #### Traits
325 
326 The queue also supports a traits template argument which defines various types, constants,
327 and the memory allocation and deallocation functions that are to be used by the queue. The typical pattern
328 to providing your own traits is to create a class that inherits from the default traits
329 and override only the values you wish to change. Example:
330 
331  struct MyTraits : public moodycamel::ConcurrentQueueDefaultTraits
332  {
333  static const size_t BLOCK_SIZE = 256; // Use bigger blocks
334  };
335 
336  moodycamel::ConcurrentQueue<int, MyTraits> q;

338 #### How to dequeue types without calling the constructor
339 
340 The normal way to dequeue an item is to pass in an existing object by reference, which
341 is then assigned to internally by the queue (using the move-assignment operator if possible).
342 This can pose a problem for types that are
343 expensive to construct or don't have a default constructor; fortunately, there is a simple
344 workaround: Create a wrapper class that copies the memory contents of the object when it
345 is assigned by the queue (a poor man's move, essentially). Note that this only works if
346 the object contains no internal pointers. Example:
347 
348  struct MyObjectMover
349  {
350  inline void operator=(MyObject&& obj)
351  {
352  std::memcpy(data, &obj, sizeof(MyObject));
353 
354  // TODO: Cleanup obj so that when it's destructed by the queue
355  // it doesn't corrupt the data of the object we just moved it into
356  }
357 
358  inline MyObject& obj() { return *reinterpret_cast<MyObject*>(data); }
359 
360  private:
361  align(alignof(MyObject)) char data[sizeof(MyObject)];
362  };

407 ## Diving into the code
408 
409 If you're interested in the source code itself, it helps to have a rough idea of how it's laid out. This
410 section attempts to describe that.
411 
412 The queue is formed of several basic parts (listed here in roughly the order they appear in the source). There's the
413 helper functions (e.g. for rounding to a power of 2). There's the default traits of the queue, which contain the
414 constants and malloc/free functions used by the queue. There's the producer and consumer tokens. Then there's the queue's
415 public API itself, starting with the constructor, destructor, and swap/assignment methods. There's the public enqueue methods,
416 which are all wrappers around a small set of private enqueue methods found later on. There's the dequeue methods, which are
417 defined inline and are relatively straightforward.
418 
419 Then there's all the main internal data structures. First, there's a lock-free free list, used for recycling spent blocks (elements
420 are enqueued to blocks internally). Then there's the block structure itself, which has two different ways of tracking whether
421 it's fully emptied or not (remember, given two parallel consumers, there's no way to know which one will finish first) depending on where it's used.
422 Then there's a small base class for the two types of internal SPMC producer queues (one for explicit producers that holds onto memory
423 but attempts to be faster, and one for implicit ones which attempt to recycle more memory back into the parent but is a little slower).
424 The explicit producer is defined first, then the implicit one. They both contain the same general four methods: One to enqueue, one to
425 dequeue, one to enqueue in bulk, and one to dequeue in bulk. (Obviously they have constructors and destructors too, and helper methods.)
426 The main difference between them is how the block handling is done (they both use the same blocks, but in different ways, and map indices
427 to them in different ways).
428 
429 Finally, there's the miscellaneous internal methods: There's the ones that handle the initial block pool (populated when the queue is constructed),
430 and an abstract block pool that comprises the initial pool and any blocks on the free list. There's ones that handle the producer list
431 (a lock-free add-only linked list of all the producers in the system). There's ones that handle the implicit producer lookup table (which
432 is really a sort of specialized TLS lookup). And then there's some helper methods for allocating and freeing objects, and the data members
433 of the queue itself, followed lastly by the free-standing swap functions.
